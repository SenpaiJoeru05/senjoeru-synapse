# 1. Overview

Phase 1 establishes the persistent data foundation of SenJoeru Synapse.

The current implementation relies heavily on JSON files generated by collectors for runtime information. While this approach works well for live monitoring, it is not suitable for managing application-owned data such as projects, tasks, repositories, settings, analytics, or historical information.

The purpose of this phase is to introduce a proper persistence layer using SQLite while preserving the existing collector architecture.

This phase intentionally avoids introducing new AI capabilities.

No prompt optimization, orchestration, autonomous workflows, or token optimization will be implemented during this phase.

The objective is to create a reliable backend that future phases can safely build upon.

# 7. Current Architecture (As-Is)

The current implementation successfully provides real-time monitoring through a collector-driven architecture.

Collectors observe Claude Runtime, repositories, and Git activity, then generate JSON files that are consumed by the backend and frontend.

Current data flow:

```

Claude Runtime
        │
        ▼
Collectors
        │
        ▼
metrics/*.json
        │
        ▼
Backend API
        │
        ▼
WebSocket
        │
        ▼
Mission Control

```

This architecture provides excellent real-time performance but introduces several long-term limitations.

Current limitations include:

- JSON files acting as persistent storage
- Difficult relationship queries
- Limited historical data
- Duplicate representations of information
- Increasing maintenance complexity as features grow

These limitations motivate the introduction of SQLite while preserving the existing collector architecture.

# 15. Database Architecture

SQLite becomes the permanent source of truth for every business entity owned by SenJoeru Synapse.

The database is designed around business domains rather than application screens.

Each table represents a single business entity with a clearly defined responsibility.

Relationships between entities should be explicit and normalized.

The database should remain simple, maintainable, and easily extensible.

---

# Database Principles

The following principles govern database design.

## Principle 1

Each table represents one business entity.

Avoid combining unrelated concepts into a single table.

---

## Principle 2

Relationships should be explicit.

Foreign keys should be used whenever possible.

---

## Principle 3

Runtime data should never become permanent data unless intentionally synchronized.

---

## Principle 4

Historical information should be append-only whenever practical.

Avoid overwriting valuable engineering history.

---

## Principle 5

Every table should include standard audit fields.

Recommended fields:

- id
- created_at
- updated_at

Soft deletes may be introduced in future phases if required.

## Repository Module

Purpose

Manage repositories connected to projects.

- git branch
- repository name

Repository contents remain on the filesystem.

---

## Task Module

Purpose

Track engineering work.

Stores:

- title
- description
- priority
- status
- assigned agent
- project
- repository

Tasks become first-class business entities.

Source & ownership (important):

- Tasks are **authored by Claude agents** in `C:\Users\<user>\.claude\tasks.json`,
  exactly as they are today (per the global `CLAUDE.md` rule). This does not change.
- `.claude/tasks.json` is an **external input feed owned by Claude** — Synapse only
  READS it and MUST NEVER write to it.
- The Task Collector reads `tasks.json`; the **SynchronizationService** imports each
  change into the SQLite `tasks` table (upsert by task `id`).
- The SQLite `tasks` table is the **permanent source of truth** for Synapse. It keeps
  full history even after entries change or disappear from `tasks.json`.
- Writes flow one direction only (Claude → `tasks.json` → SQLite → UI), so there is no
  dual ownership. See ARCHITECTURE-V2 "Sanctioned exception: Tasks".

---

## Workspace Module

Purpose

Manage local workspaces.

Examples

Development

Personal

Experimental

Client Projects

Future versions may support multiple workspaces.

---

## Settings Module

Purpose

Persist application configuration owned by Synapse.

Examples

- Theme
- Polling Interval
- Monitored Repositories list
- Budget limits (hourly / weekly)
- Default Workspace
- Feature Flags

Persistent application settings should live in SQLite once available (they are
NOT runtime cache — deleting them would lose real configuration).

Excluded — presentation state stays in the frontend:

- Window Layout, open tabs, expanded panels, active filters, selected repository
  remain frontend/browser state per ARCHITECTURE-V2 (Presentation Ownership).
  They must NOT be stored in the SQLite settings table.

Collector configuration:

- Some settings (Polling Interval, Monitored Repositories) are needed by the
  Collector Engine, which sits below the Data Layer and must not read SQLite
  directly. The backend/SettingsService supplies this operational config to the
  collector at startup and on change (e.g. via startup arguments or an internal
  read-only endpoint) — the collector never queries the database itself.

---

## Analytics Module

Purpose

Store long-term engineering metrics.

Examples

- Token usage
- Cost history
- Agent activity summaries
- Repository statistics

Analytics are historical records rather than live metrics.

---

## Execution History Module

Purpose

Record completed engineering activity.

Examples

- Task completed
- Agent execution
- Git commit linked to task
- Session summary

Execution history supports future reporting and analytics.

Relationship to the runtime activity feed (avoid duplication):

- The live `activity.json` feed is **runtime cache** — a rolling, disposable view
  of "what is happening now" (regenerated every poll, capped, rebuildable).
- `execution_history` is the **permanent, append-only** record of
  business-significant completed events, written once by a service.
- They are NOT two copies of the same thing: the activity feed is ephemeral
  display data; execution history is durable business history. A service decides
  which runtime events are significant enough to persist into `execution_history`.

# 21. Synchronization Strategy

Collectors continue generating runtime metrics.

Application Services decide whether runtime information should become persistent.

Example

```

Collector

↓

Runtime Event

↓

Application Service

↓

Validation

↓

SQLite

↓

WebSocket

```

Collectors never insert records directly into SQLite.

This responsibility belongs to the Application Layer.

Task ingestion (concrete example):

```

Claude agent edits .claude/tasks.json

↓

Task Collector detects change (read-only)

↓

SynchronizationService validates + upserts by task id

↓

SQLite tasks table (permanent record + history)

↓

WebSocket → Mission Control

```

Synapse never writes `.claude/tasks.json`. If SQLite and `tasks.json` ever
disagree, `tasks.json` is the latest author intent and SQLite is reconciled FROM
it — never the reverse.

The existing Collector Engine should remain largely unchanged.

Collectors continue observing:

- Claude Runtime
- Git
- Filesystem
- Runtime Sessions

Collectors continue generating:

- Metrics
- Activity
- Runtime State

The only architectural change is where the data flows afterward.

Previous flow:

```

Collector

↓

JSON

↓

Frontend

```

New flow:

```

Collector

↓

Runtime Cache

↓

Synchronization Service

↓

SQLite (if required)

↓

WebSocket

↓

Frontend

```

Collectors remain observers.

They never own business logic.

Clarification on "unchanged": the collectors' observation behavior is unchanged
(including reading `.claude/tasks.json`). What is removed is only their role as
the *store of record* — they no longer treat `metrics/tasks.json` as the
authoritative task store. Persisting tasks into SQLite becomes the
SynchronizationService's job. This is a change in *destination*, not in what the
collector observes.

# Phase 1 Summary

At the conclusion of Phase 1, SenJoeru Synapse transitions from a runtime-monitoring application into a platform with a true application data layer.

SQLite becomes the permanent source of truth for all business entities owned by Synapse.

Collectors continue serving as lightweight observers of runtime systems, preserving the existing zero-token monitoring architecture.

The backend adopts a service-oriented design with clear ownership boundaries, enabling future phases to introduce AI orchestration, engineering knowledge management, context optimization, and token optimization without requiring significant architectural changes.

Phase 1 establishes the technical foundation upon which all subsequent capabilities of SenJoeru Synapse will be built.
